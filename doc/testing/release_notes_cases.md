# Release Notes Cases

Concrete before/after examples of what GitHub Release notes look like at each stage of the deployment pipeline.

> **Design:** "Deployment History" — the `## 📦 Deployment History` section is **written once** and **only grows**. `consolidate-changelog` appends a `⏳ Pending` bullet for the next environment (preserving all existing `✅` bullets). `mark-deployed` replaces that `⏳` bullet in-place with a `✅` timestamp. History is never lost.

---

## How Release Notes Evolve

Each release starts with notes generated by semantic-release. As it moves through environments the `## 📦 Deployment History` section at the bottom grows one bullet at a time.

```
[Semantic-release notes]
         │
         ▼  consolidate-changelog (stage)  ← creates section with ⏳ bullet
[Semantic-release notes]
---
## 📦 Deployment History

- ⏳ Pending deployment to Staging | vOLD → vNEW

[changelog content]
         │
         ▼  mark-deployed (stage)  ← replaces ⏳ with ✅ in-place
[Semantic-release notes]
---
## 📦 Deployment History

- ✅ Deployed to Staging | vOLD → vNEW — <timestamp>

[changelog content]
         │
         ▼  consolidate-changelog (prod)  ← preserves ✅, appends new ⏳
[Semantic-release notes]
---
## 📦 Deployment History

- ✅ Deployed to Staging | vOLD → vNEW — <timestamp>
- ⏳ Pending deployment to Production | vOLD → vNEW

[changelog content]
         │
         ▼  mark-deployed (prod)  ← replaces ⏳ with ✅ in-place
[Semantic-release notes]
---
## 📦 Deployment History

- ✅ Deployed to Staging | vOLD → vNEW — <timestamp>
- ✅ Deployed to Production | vOLD → vNEW — <timestamp>

[changelog content]
```

## Case 1 — Normal Progression (stage then prod)

**Setup:** v1.3.0 released. v1.2.0 was last deployed to both stage and prod.

### Step 0 — Semantic-release notes (no deployment yet)

```markdown
## [1.3.0](https://github.com/org/repo/compare/v1.2.0...v1.3.0) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
* add export to CSV

### 🐛 Bug Fixes
* fix null error in order aggregation
```

### Step 1 — After `consolidate-changelog` runs for stage (pending)

```markdown
## [1.3.0](https://github.com/org/repo/compare/v1.2.0...v1.3.0) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
* add export to CSV

### 🐛 Bug Fixes
* fix null error in order aggregation

---

## 📦 Deployment History

- ⏳ Pending deployment to Staging | v1.2.0 → v1.3.0

## [1.3.0](https://github.com/org/repo/compare/v1.2.0...v1.3.0) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
* add export to CSV

### 🐛 Bug Fixes
* fix null error in order aggregation
```

> This is the state the approver sees before clicking Approve.

### Step 2 — After `mark-deployed` runs for stage

`- ⏳ Pending deployment to Staging` is replaced in-place:

```markdown
## [1.3.0](https://github.com/org/repo/compare/v1.2.0...v1.3.0) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
* add export to CSV

### 🐛 Bug Fixes
* fix null error in order aggregation

---

## 📦 Deployment History

- ✅ Deployed to Staging | v1.2.0 → v1.3.0 — 2026-04-24 14:05 UTC

## [1.3.0](...)
...
```

### Step 3 — After `consolidate-changelog` runs for prod

Existing `✅` bullet is preserved; new `⏳` bullet is appended:

```markdown
## [1.3.0](https://github.com/org/repo/compare/v1.2.0...v1.3.0) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
* add export to CSV

### 🐛 Bug Fixes
* fix null error in order aggregation

---

## 📦 Deployment History

- ✅ Deployed to Staging | v1.2.0 → v1.3.0 — 2026-04-24 14:05 UTC
- ⏳ Pending deployment to Production | v1.2.0 → v1.3.0

## [1.3.0](...)
...
```

> Stage history is preserved — no longer lost when prod runs.

### Step 4 — After `mark-deployed` runs for prod

`- ⏳ Pending deployment to Production` is replaced in-place:

```markdown
## [1.3.0](https://github.com/org/repo/compare/v1.2.0...v1.3.0) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
* add export to CSV

### 🐛 Bug Fixes
* fix null error in order aggregation

---

## 📦 Deployment History

- ✅ Deployed to Staging | v1.2.0 → v1.3.0 — 2026-04-24 14:05 UTC
- ✅ Deployed to Production | v1.2.0 → v1.3.0 — 2026-04-28 09:15 UTC

## [1.3.0](...)
...
```

## Case 2 — First Deployment to Environment

**Setup:** No previous successful deployments to stage (or prod).

### After `consolidate-changelog` (pending)

```markdown
### 🚀 Features
* initial release

---
```

### After `mark-deployed`

```markdown
## [1.0.0](https://github.com/org/repo/releases/tag/v1.0.0) (2026-04-10)

### 🚀 Features
* initial release

---

## 📦 Deployment History

- ✅ Deployed to Staging | Initial Deployment — 2026-04-10 11:00 UTC

This is the first deployment to Staging.
```

## Case 3 — Skip Versions

**Setup:** v1.0.0 was last deployed to stage. v1.1.0, v1.2.0, and v1.3.0 have been released but not deployed. Now deploying v1.3.0 to stage.

### After `consolidate-changelog` (pending)

```markdown
## [1.3.0](...) (2026-04-24)

### 🚀 Features
* feature in 1.3.0

---

## 📦 Deployment History

- ⏳ Pending deployment to Staging | v1.0.0 → v1.3.0

## [1.3.0](...) (2026-04-24)

### 🚀 Features
* feature in 1.3.0

## [1.2.0](...) (2026-04-20)

### 🐛 Bug Fixes
* fix in 1.2.0

## [1.1.0](...) (2026-04-15)

### 🚀 Features
* feature in 1.1.0
```

> All three skipped versions are consolidated into a single section — the approver sees everything at once.

## Case 4 — Redeploy Same Version

**Setup:** v1.3.0 was already deployed to stage. Someone triggers stage deploy for v1.3.0 again.

### After `consolidate-changelog`

The existing `✅ Deployed to Staging` bullet is preserved; a new `⏳` bullet is appended:

```markdown
## [1.3.0](...) (2026-04-24)

### 🚀 Features
* add new reporting dashboard

---

## 📦 Deployment History

- ✅ Deployed to Staging | v1.2.0 → v1.3.0 — 2026-04-24 14:05 UTC
- ⏳ Pending deployment to Staging | Redeploying v1.3.0

No new changes — redeploying v1.3.0.
```

### After `mark-deployed` (redeploy)

```markdown
## 📦 Deployment History

- ✅ Deployed to Staging | v1.2.0 → v1.3.0 — 2026-04-24 14:05 UTC
- ✅ Deployed to Staging | Redeploying v1.3.0 — 2026-04-25 10:30 UTC

No new changes — redeploying v1.3.0.
```

> Each redeploy appends a new entry — full history is preserved.

## Case 5 — Stage Deployed, Prod Never Runs

**Setup:** v1.3.0 is deployed to stage but prod deployment is never triggered (or is declined). v1.4.0 is the next regular release.

### v1.3.0 release notes — final state (prod never ran)

```markdown
## [1.3.0](...) (2026-04-24)

### 🚀 Features
* add new reporting dashboard

---

## 📦 Deployment History

- ✅ Deployed to Staging | v1.2.0 → v1.3.0 — 2026-04-24 14:05 UTC

## [1.3.0](...)
...
```

> v1.3.0 keeps the `✅ Deployed to Staging` bullet permanently. Because prod was never triggered, no prod entry is ever added.

### When v1.4.0 deploys to prod — what version range is used?

Prod's last successful deployment was v1.2.0. `consolidate-changelog` finds v1.2.0 and consolidates v1.3.0 + v1.4.0 changes into v1.4.0's release notes:

```markdown
## 📦 Deployment History

- ⏳ Pending deployment to Production | v1.2.0 → v1.4.0

## [1.4.0](...) (2026-04-28)

### 🚀 Features
* new feature in 1.4.0

## [1.3.0](...) (2026-04-24)

### 🚀 Features
* add new reporting dashboard
```

> v1.3.0's content appears inside v1.4.0's release notes — the skipped version is accounted for.

## Case 6 — Hotfix Deployment

**Setup:** v1.3.0 is in prod. A critical bug is found. Hotfix branch creates `v1.3.1-hotfix-from-v1-3-0.1`. It deploys directly to prod.

### After `consolidate-changelog` (pending)

```markdown
## [1.3.1-hotfix-from-v1-3-0.1](...) (2026-04-24)

### 🐛 Bug Fixes
* fix critical null pointer in AAR processing

---

## 📦 Deployment History

- ⏳ Pending deployment to Production | v1.3.0 → v1.3.1-hotfix-from-v1-3-0.1

## [1.3.1-hotfix-from-v1-3-0.1](...) (2026-04-24)

### 🐛 Bug Fixes
* fix critical null pointer in AAR processing
```

### After `mark-deployed` + `promote-release` skipped

```markdown
## [1.3.1-hotfix-from-v1-3-0.1](...) (2026-04-24)

### 🐛 Bug Fixes
* fix critical null pointer in AAR processing

---

## 📦 Deployment History

- ✅ Deployed to Production | v1.3.0 → v1.3.1-hotfix-from-v1-3-0.1 — 2026-04-24 16:22 UTC

## [1.3.1-hotfix-from-v1-3-0.1](...)
...
```

> The release stays as a **prerelease** — `promote-release` detects "hotfix" in the tag and skips the `--latest` flag.

## Case 7 — Regular Release After Hotfix (Divergent Lineages)

**Setup:** v1.3.1 hotfix was the last prod deployment. v1.4.0 is cut from main (which was branched before the hotfix existed). v1.4.0's `CHANGELOG.md` does **not** contain `[1.3.1]`.

### What happens

The awk script walks `v1.4.0`'s `CHANGELOG.md` looking for `## [1.3.1]` as the stopping point — it never finds it, so it captures everything from `[1.4.0]` down to end of file.

### After `consolidate-changelog` for prod

```markdown
## [1.4.0](...) (2026-04-28)

### 🚀 Features
* new feature on main

---

## 📦 Deployment History

- ⏳ Pending deployment to Production | v1.3.1-hotfix-from-v1-3-0.1 → v1.4.0

## [1.4.0](...) (2026-04-28)

### 🚀 Features
* new feature on main

## [1.3.0](...) (2026-04-20)

### 🚀 Features
* add new reporting dashboard

## [1.2.0](...) (2026-04-15)

...all earlier versions...
```

> The section is larger than expected because the hotfix stopping point wasn't found. This is acceptable — it shows everything in v1.4.0's lineage. The hotfix fix must be backmerged to main via a separate commit to appear cleanly in a future release.

## Scenario Summary Table

| Scenario | Last Deployed | Deploying | OLD in CHANGELOG? | Consolidated Content |
|----------|---------------|-----------|-------------------|----------------------|
| Normal | v1.2.0 | v1.3.0 | ✅ Yes | v1.3.0 changes only |
| Skip versions | v1.0.0 | v1.3.0 | ✅ Yes | v1.1.0 + v1.2.0 + v1.3.0 |
| Redeploy | v1.3.0 | v1.3.0 | ✅ Yes | "No new changes" |
| First deploy | _(none)_ | v1.0.0 | N/A | "Initial Deployment" |
| Hotfix | v1.3.0 | v1.3.1 (hotfix) | ✅ Yes | v1.3.1 hotfix changes |
| After hotfix (divergent) | v1.3.1 (hotfix) | v1.4.0 (main) | ❌ No | All v1.4.0 lineage |
| Stage-only, skip to v1.4.0 | v1.2.0 (prod) | v1.4.0 | ✅ Yes | v1.3.0 + v1.4.0 |
